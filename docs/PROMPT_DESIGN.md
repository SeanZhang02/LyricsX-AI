# Prompt 设计 + 输出格式契约

> 现行 prompt 的**唯一真源**在 `AppController.swift` 的 `buildPrompt`(硬编码,目标语言写死简体中文)。本文档只记设计理念与实证依据,不复制全文,避免与代码漂移。

## 定位

一段英文写就的「文学歌词译者」系统 prompt,目标语言写死 **简体中文**(用户实际只把英/西/葡歌译成中文;`targetCode` 仍用于给译文打 `.translation(languageCode:)` attachment tag,与 prompt 文本解耦)。temperature 0。默认模型 opus-4.8,失败按链 fallback(见 `requestTranslation`)。

## 设计原则

1. **行对齐 > 一切**:`n|译文` 逐行协议,不用 JSON——错位可检测、可逐行恢复、免疫模型在 JSON 里输出未转义字符/额外废话。译文禁含换行、`[`、`]`、`|`。
2. **第 i 行配到第 j 行译文比没有译文更糟**:所以有覆盖率门控 + 内容校验(HARDENING H2/H3),prompt 只是第一道防线。
3. **版权拒译是真实风险**:prompt framing 为「用户本地歌词文件 / 仅个人查看」,降低拒译概率;拒译检测兜底在解析层。
4. **先通读全篇,再逐行**:prompt 第一步强制模型在动笔前形成对整首歌的完整理解(主旨 / 情绪弧线 / 说话人 / 语体 / 复现意象与副歌),此理解只作内部脚手架,绝不出现在输出。副歌用词在通读后统一并复用。
5. **优先级显式化 + 平局规则**:五条优先级(输出契约 > 保义与对齐 > 自然连贯的中文 > 意象/声音/情绪/语体 > 韵律与适度凝练),"冲突时高优先级胜"。这条是让一段长 prompt 不自相矛盾的关键。

## "信达雅" 的落点(本 prompt 的核心立场)

歌词翻译的「雅」**不来自擅自堆砌华丽意象**,只来自三件事,prompt 把每一件都写成了可执行规则:

1. **重建中文句法**——不逐字直译,按中文语法重排;相邻行构成一个语法/诗意单元时,作为一句连续中文分布到对应各行(`Line mapping and cross-line syntax` 段)。
2. **安排停顿与标点**——允许行尾用逗号/分号/破折号/省略号或不加标点表示句子延续;不机械地每行句号;不为求短而删标点。
3. **让原本的意象在中文里自然相接**——保留源文隐喻与画面,用中文自然搭配落笔,不解释其「象征什么」,不把具象意象改得更美/更强/更具体。

配套的反面约束:不新增源文既无明说也无暗示的意象/信息/因果;`Elaborate only when the added words buy clarity/lyrical quality/nuance that fits the song's style, otherwise keep it tight`——给「可以更长」上刹车,防止铺张。

## 实证依据(为什么是现在这版)

对照实验:旧「凝练优先」prompt(shorter-is-better、凑四字句)vs 本候选版,同模型 opus-4.8、temp 0,在日文《今夜だけきっと》与葡文《Meditação》上逐行对比,Claude + Fable 双评。

- 结论:候选版在三条「雅」轴(句法重建 / 停顿标点 / 意象相接)全胜。旧版最严重的错误是**为凑四字句无中生有意象**(把「推落」写成「击碎」、凭空加「爱笑」)——正是「雅不该来自堆砌」所禁止的,且是其凝练偏置的结构性产物,无法靠个别改行修补。
- 在候选版基础上做了 3 处外科手术式补丁,各修一个可复现的真实缺陷:
  1. **句末语气词 / 感叹词纪律**:不把源语言句末助词(日语 ね/よ/の)逐一搬运;按语气选词,轻声呼唤不译成生硬的「喂」。
  2. **抗翻译腔**:多字直译(如 `无法相互理解`)优先换成中文母语者会写的地道说法。
  3. **铺陈刹车**:只在换来清晰/韵味且贴合整首风格时才铺陈,否则收紧;不加「的人 / 内心」这类空词或源文没有的叠词。
- 复测确认:第 3 条把简单歌(Meditação)的啰嗦压回人工润色版的紧致度,同时难歌(日文)不退步、反而更利落(自动去掉多余的「猛然」、`无法彼此理解`→`无法读懂彼此`)。

### 二轮迭代(专家评审 + 锚定实验,2026-07-23/24)

在上述版本(内部代号 MK3F)基础上,经翻译专家视角评审 + 单变量实验演进为现行版(MK6):

1. **反添加五条并一 + 补对称条款**:五条"不得添加"堆叠会诱发系统性降温(只设天花板不设地板);合并为一条并加上「降温/净化与加码同罪」(cooling or sanitizing is as unfaithful as intensifying)。
2. **性别/称谓条款(新增)**:西/葡形态性别与日语「君」的性别开放,进中文会被 他/她 强制站队;指令化为「语法未标性别的对象不得引入 他/她——用名词、你、对方或改写省代词(su voz → 那嗓音)」。弱表述版实测无效(只会换个代词),指令化表述实测命中;敬称(您)半句实测不触发,按反膨胀原则删除。
3. **传统锚定 + 三来源缰绳**:开头一句「神似 > 形似,追求化境」——单变量实测这一句能改变 84% 的行,方向大体是好的(话题前置/停顿标点/敢保超现实意象),但裸用会凭空造意象(余温/悄悄);绑死「化境只能经由句法重建、停顿标点、原有意象自然相接达成」后,四处越界全部修正且诗意保留。

### 三轮迭代(收敛,2026-07-24)——现行版

在锚定版(MK6)基础上做了一次收敛(MK10):
- **整首歌理解段新增「语言风格」**:动笔前先感受全曲内容与语言双重风格(措辞层级、口语/典雅、时代与曲风气质、句子质感),以此统领全篇节奏与用词。
- **实例凝练成规则**:9 个内嵌实例逐个处置——强化词黑名单(身体/紧紧/活不下去)改为模式表述(不注入源文未发声的躯体化/绝望/力度),解除「源文真有 hold me tight 也不敢译紧紧」的过泛化;仅保留 4 个实测证明必须靠例子才生效的微例(性别中立 su voz→那嗓音、反直译腔 わかりあえない、语气词 喂、回译查义 头也不回)。
- **去冗余/矛盾**:一行一译 4 处→1、标点自由 4 条→1、歧义 3 处→1、长度旋钮 3 处→1、静默终检 12 项→4 项。全文 2022→1770 词(−12.5%)。
- **教训**:终检砍掉锚点复查后,高难分裂句立即出现内容跨行调包(同步显示不可接受);加回一条「每行核心意象/动作留在本行,相邻行可共语法、绝不换内容」后归位。收敛回归 11/12 焦点检查通过(唯一失败项为已知的人工截断词摇摆用例),普通歌行均长度漂移 <2%。
4. **已测未采纳**:精简 31% token 的 MK4LEAN(质量不降但用户偏好完整版);「直译草稿→独立润色」任务重构 MK7(Lost in Literalism ACL 2025 的机制,在我们已低翻译腔的基线上无可测增益——天花板效应,留待真实歌曲出现翻译腔时复测)。
5. **评测方法教训**:廉价模型(deepseek)做 LLM-judge 有严重位置偏差(15 票 14 票投 A 位);必须换位一致性过滤,过滤后对微调级差异无信号——judge 只在差距大时可作参考,微差场景以人工细读为准。

版本全文与逐行对照存于本地 `prompt-eval/`(gitignored,含增量评测脚本,老版本译文全缓存)。

## 输出格式契约(实现对照解析层 + HARDENING H3/H4)

1. 逐行匹配 `编号|译文`(兼容全角 `｜`);不匹配的行(解释 / ``` / `<think>` 泄漏)直接忽略。
2. 重复编号**末个生效**(模型先 echo 输入再输出译文时,首个生效会把原文当译文收下)。
3. `-` 与 译文==原文 → 跳过该行(不写 attachment,该行显示层自动只显原文)。
4. `n|-` 仅用于:空行 / 本身已是简体中文的行 / 纯制作信息行 / 纯无词性发声(拟声、ad-lib)。有意义的感叹词、副歌、混合语言行**不得**用 `-`。
5. sanitize 见 HARDENING H4;覆盖率分层门控见 HARDENING H2。

## 成本/延迟参考

- 典型歌 30-80 行、每行 <60 字符;`行号|原文` 打包 + prompt 约 3-5k input tokens(prompt 本身较长,换取质量),输出约 1-2.5k。
- opus-4.8 质量最好、单价最高;deepseek-v3.1 性价比最高、可作 fallback。落盘缓存后二次播放 $0。
- 延迟 10-60s 可接受(歌不中断,译文到了当场上屏);超时上限见 `performRequest`。
